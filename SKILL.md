# 小米辣 · 个股深度分析框架 v2.7.0

> **本文件是 Skill 入口。** 完整分析框架见 [`references/framework.md`](references/framework.md)（§一~§十三）。GP-A交叉验证规范见 [`references/gp-a-cross-validation.md`](references/gp-a-cross-validation.md)。

## Overview

A股个股深度分析的标准化流水线。将分析拆解为13个独立模块，核心差异化：三视角交叉验证暴露盲区并量化置信度、行业权重调节避免一刀切评分、特殊标的适配层覆盖双主业/ST/亏损/次新/周期五类边缘场景。

分析产物为 Markdown 全文 + 暗色金融终端风格 H5 页面。数据采集由 `scripts/` 下6个脚本驱动。

---

## When to Use

**触发条件**（任一即加载）：
- "分析下XX股票 / 帮我看看XX / XX怎么样"
- "XX行业怎么看 / XX板块行情"
- "这几只哪个好 / XX vs YY vs ZZ"
- "最近XX跌了好多 / XX涨疯了"

**不要用这个 skill 的场景**（反触发）：
- 用户只是随口问股价/PE/市值 → 直接调 `scripts/fetch_quote.py` 回答，不加载全框架
- 用户问的是港美股/加密货币/期货 → 本框架仅覆盖A股
- 用户问量化策略/回测/因子 → 加载 `quant-kb` skill
- 用户问的是郑希/基金经理观点 → 加载 `zhengxi-views` skill

## 执行步骤

每个步骤括号内为**完成标准**——满足后才可进入下一步。

```
1. 加载 references/framework.md
   （完成标准：确认§一~§十三结构清晰，数据源表已读取）

2. 执行§二"研报解读" → 读取年报/季报 → 提取核心事实 → 判断标的类型 → 应用§十二特殊标的适配
   ⚠️ 必须先读年报/季报的"主营业务构成"章节，以官方披露的业务结构为准
   ⚠️ 不要依赖东财F10的行业标签——标签可能滞后，实际业务可能已转型
   ⚠️ §二是后续所有板块的事实锚点，不得跳过、不得用F10摘要替代
   💡 v2.4.1推荐：年报/季报提取优先使用 delegate_task（toolsets=["browser","terminal","web"]），子代理并行爬取cninfo.com.cn/巨潮资讯网，比主代理单线程浏览器快5-10倍
   （完成标准：§二输出格式完整填写 + 标的类型"正常/ST/亏损/次新/周期/双主业"）

3. 数据采集（6个脚本按顺序执行）：
   a. python3 scripts/fetch_quote.py CODE --cross-check
      ⚠️ v2.7.1：必须启用--cross-check。GP-A周期涨跌由K线独立核算交叉验证，偏差>5pp自动替换。
      （完成标准：gp_cross_check.status已检查。若fail→gp_a_*已被K线校准值替换，全报告使用替换值）
   b. python3 scripts/fetch_kline.py CODE --days=250
      （完成标准：返回 ≥200 条K线，MA20/MA60 已计算）
   c. python3 scripts/fetch_finance.py CODE → browser执行提取
      （完成标准：获取到营收/净利/毛利率/ROE/ROA/资产负债率至少4项）
   d. python3 scripts/fetch_holders.py CODE → browser执行提取
      （完成标准：前十大股东+基金持仓趋势已获取）
   e. python3 scripts/fetch_fund_flow.py CODE → browser执行提取
      （完成标准：北向净流入趋势+融资余额变化已获取，或标注⚠️数据缺失）
   f. python3 scripts/fetch_announce.py CODE --days=90 → browser执行提取
      （完成标准：近3月公告列表≥5条，含日期+标题+类型）

4. 按§一~§八顺序整理数据并逐段输出分析
   （完成标准：每段末尾均有「📎 信息来源」标注）

5. §五.5极端行情诊断检查
   （完成标准：明确输出"触发/未触发"，触发则完成诊断清单全部✅/❌标记）

6. §十视角交叉验证
   （完成标准：多头有定价错误证据 ≥1条，空头有极限亏损测算 ≥1条，审计有数据矛盾 ≥1处 + 管理层展望vs实际动作核对）

7. 数据缺失检查 → 应用第七条铁律
   （完成标准：所有缺失字段已显式标注⚠️，受影响板块评分已打折或置信度已降级）

8. 代入§十一行业权重调节 → 计算基础评分
   （完成标准：6维加权分已填入§九.1评分表，基础分__/50）

9. 应用§十置信度系数修正 → 最终评分
   （完成标准：最终分 = 基础分 × 系数，已填入§九.3）

10. Markdown全文输出 → H5页面渲染
    （完成标准：H5包含所有必备要素+§十三免责声明在末尾）
```

## ⛔ 铁律（10条）

### 第一条：数据来源红线
- ✅ 财报：上交所/深交所公告、东财F10、巨潮资讯网
- ✅ 政策：国务院/发改委/工信部/证监会官网、新华社、东财转载官媒
- ✅ 行情：腾讯API `qt.gtimg.cn` → 降级 东财个股页浏览器提取
- ❌ 禁自媒体作财报/政策来源（微信公众号/雪球个人号/微博大V/抖音快手）
- ⚠️ 大V观点可引用但须标注"来源：XX，非官方数据"
- ⚠️ 股吧/雪球声量指标仅作情绪参考，不取其中具体信息

### 第二条：溯源标注
- 每段末尾列「📎 信息来源」，正文角标 `[¹]` 与来源一一对应
- 无来源标记的结论 = 不可引用

### 第三条：时效性门槛
- 财报用最新一期，标注截止日期；研报取近3月内；政策标注发文日期
- 过期政策标"已废止"或"已被XX取代"

### 第四条：利益相关标注
- 保荐/主承销商研报标 ⚠️利益相关；重仓机构估值观点标 ⚠️持仓方

### 第五条：可比公司选取标准
- 同板块至少5只上限8只；以**年报主营业务构成**为产品线对标依据
- 东财F10行业标签仅作参考，不取代年报业务结构分析
- 市值相近（±50%）或产品结构相似；亏损股单独分组

### 第六条：三视角交叉验证
- 多头≥1条定价错误证据，空头≥1条极限亏损测算，审计≥1处数据矛盾
- 不达标 = 重做。三次不达标 → 置信度强制🔴

### 第七条：数据缺失透明化
- 缺失显式标注⚠️，不得用0/空值/估计值填充
- 核心指标缺失超3个 → 该板块评分×0.7
- 缺失致无法交叉验证 → 置信度强制🔴

### 第八条：跨标的/跨分析数据隔离 ⬅️ v2.3.2新增
- 每只标的的数据采集独立执行，不从上一只标的的context中复用任何财务数据
- 分析新标的时，行情/K线/财务全部通过scripts重新拉取
- 若§三的净利润、毛利率等数据（或§二关键财务数据）与之前分析的标的雷同 → 触发自检：数据是否串用了？
- 每次§一输出前强制核对：本标的证券代码是否为当前分析标的？

### 第九条：跨skill建议一致性 ⬅️ v2.5.0新增
- 同一标的、同一时间点、不同skill的分析结论**不能彼此矛盾**
- 例：禁止Framework建议"15%仓位分3批建仓"而DSA Dashboard建议"0%观望"
- 若Framework和DSA Dashboard同时被加载分析同一标的（如用户对比两个skill），执行步骤：
  1. 先独立完成各自分析
  2. 在输出结论前交叉核对两个skill的信号方向（买入/观望/卖出）是否一致
  3. 若不一致 → 在各自的报告中显式标注"⚠️本报告与{skill名}建议不一致"并解释分歧原因
  4. 分歧原因必须具体（如"DSA偏技术面所以见MA60下方即观望，Framework偏估值面所以认为PE18倍低估可建仓"）
- 目的：用户对比两个skill时，看到的是"有解释的分歧"而非"自相矛盾的产品"

### 第十条：GP-A周期涨跌交叉验证 ⬅️ v2.7.1新增
- 腾讯API的GP-A四段涨跌幅（fields[62]-[65]）为预计算值，存在更新滞后/计算异常风险
- **实盘案例**：巨化股份 600160，API近3月报 -3.14%，K线实算 +48.2%（偏差 51pp），近6月 API报 +1.13% 实算 +32.77%（偏差 31pp）
- **必须启用** `fetch_quote.py --cross-check`，由K线独立计算四段涨跌幅并交叉验证
- 偏差 > 5个百分点 → 自动替换为K线计算值，全报告使用替换后的 gp_a_month/gp_a_3month/gp_a_6month/gp_a_year
- 若K线数据不可得（unverified状态） → 标注 ⚠️GP-A未交叉验证，但继续使用API原始值

## Common Pitfalls

1. **凭旧印象判断行业，不读年报业务结构。** 孚日股份东财F10标"纺织服装"，但实际VC锂电池电解液添加剂是核心增长极——若不读年报就按家纺股对标，PE比较结论完全失真。正确做法：步骤2强制执行年报主营业务构成提取，以官方披露为准。

2. **F10行业标签滞后导致对标错误。** 东财的行业分类可能未反映最新业务转型。例：标的从传统制造转型新能源材料，标签仍是"制造业"。正确做法：§一做完后搜索近3月公告确认主营业务是否发生重大变化，若发现新业务板块以新业务为对标依据。

3. **数据采集不完整就开写。** 正确做法：提取失败用降级方案，再失败标注⚠️数据缺失。

4. **东财F10股东/基金页SPA动态加载提取失败。** 正确做法：优先用browser_snapshot(full=true)全文搜索提取；降级从页面全文用正则提取关键数字；再失败标注⚠️。

5. **交叉验证走过场。** 正确做法：多头必须有具体证据，空头必须算数字，审计必须找矛盾。

6. **PE跨行业比。** 正确做法：严格按第五条铁律选可比公司。

7. **忽略特殊标的适配层。** 正确做法：步骤2先判定类型，§十二有明确指引。

8. **忘了免责声明。** 正确做法：H5 footer 固定输出§十三。

9. **H5板块顺序搞反。** 即使分析流程中§二（研报解读）是事实锚点优先读取，H5页面必须严格按照编号顺序排列（一→二→三→...）。不能因为"§二是基础"就把§二放在§一前面——编号顺序就是显示顺序。

10. **只输出Markdown忘了生成H5。** Markdown全文输出和H5页面渲染是两个独立交付物，缺一不可。Markdown放在对话中展示核心结论，H5作为完整交付件附件发送。执行步骤10"H5页面渲染"时用独立Python脚本（文章内嵌SVG走势图+CSS），写入文件后用 send_message 发送。宁德时代案例：先回了Markdown被用户追问"你是不是忘了要生成h5页面了"——这就是踩坑。

11. **§二年报/季报浏览器提取慢或失败。** ⬅️ v2.4.1：东财F10 SPA页面和巨潮资讯网的财务报告页面在浏览器中加载慢、DOM复杂。正确做法：**委托子代理提取**——用 `delegate_task`，toolsets=["browser","terminal","web"]，goal明确要求提取主营业务构成+关键财务数据+审计意见+风险因素+管理层展望。子代理可用terminal(curl)+浏览器并行爬取cninfo.com.cn，效率远高于主代理单线程浏览器操作。横店东磁案例证实该模式可在2分钟内完成全量年报+季报提取（22次API调用，132秒）。

12. **只输出Markdown忘了生成H5。** Markdown全文输出和H5页面渲染是两个独立交付物，缺一不可。Markdown放在对话中展示核心结论，H5作为完整交付件附件发送。执行步骤10"H5页面渲染"时用独立Python脚本（文章内嵌SVG走势图+CSS），写入文件后用 send_message 发送。宁德时代案例：先回了Markdown被用户追问"你是不是忘了要生成h5页面了"——这就是踩坑。

13. **腾讯API fields[44]对A+H股返回A股部分市值，不是全口径总市值。** ⬅️ v2.5.1：`qt.gtimg.cn` 的 fields[44]（总市值）对于同时A+H上市的标的仅返回A股部分市值，fields[45]才是A+H全口径总市值。纯A股标的fields[44]正确。宁德时代(300750)纯A股不受影响，但比亚迪(002594)、中芯国际(688981)等A+H股的fields[44]严重偏低。正确做法：§四同板块估值横评时，对A+H标的使用 fields[45] 作为总市值，并在数据采集时显式检查该标的是否为A+H（代码以002/688开头但港股也有上市）。宁德时代分析中比亚迪市值曾误用3085亿（fields[44]），实际应为8066亿（fields[45]），差距2.6倍。

14. **H5生成脚本中同行对比表硬编码数据。** ⬅️ v2.6.1：§四"同板块估值横评"的所有可比公司数据（PE/PB/市值/涨跌幅）不得在Python脚本或HTML模板中手写——必须通过 `scripts/fetch_quote.py` 批量拉取。宁德时代H5生成脚本(gen_catl_h5.py)曾将比亚迪市值硬编码为3085亿，虽然最终HTML被手动修正为8066亿，但脚本源码仍含错误值，下次重新生成会带入。正确做法：H5生成脚本中§四表格通过 `subprocess.run(['python3', 'scripts/fetch_quote.py', 'BYD_CODE', 'EVE_CODE', ...])` 批量拉取后动态填入，零硬编码。`scripts/fetch_quote.py` 已内置 fields[45] 优先逻辑，可正确处理A+H标的。

15. **GP-A周期涨跌幅不可盲信。** ⬅️ v2.7.1：腾讯API的GP-A四段涨跌幅存在更新滞后/计算异常问题——巨化股份(600160)实测API近3月报-3.14%，K线实算+48.2%，偏差高达51个百分点。近6月和近1年同样严重失准。**正确做法**：`fetch_quote.py` 必须带 `--cross-check`，自动从ifzq K线独立计算四段涨跌幅并交叉验证。偏差>5pp自动替换为K线计算值。不可在任何分析报告中直接使用未验证的GP-A原始值。
